## หอคอยบั๊ก 12 ชั้น

ไม่ใช่บั๊กเดียวที่ซับซ้อน แต่คือ **12 บั๊กคนละสาเหตุที่แสดงอาการเหมือนกันบนจอ**
ทุกชั้นมองเห็นเป็น "จอดำ" หรือ "จอโมเสก" เหมือนกันหมด
แก้ชั้นหนึ่งเสร็จ ชั้นถัดไปก็โผล่มาหน้าตาเหมือนเดิม:

| ชั้น | อาการบนจอ | สาเหตุจริง |
|------|-----------|-----------|
| 1 | ไม่มี pixel | Bus protocol ผิด — AXS15231 เป็น QSPI แบบไม่มี DC pin |
| 2 | โมเสก | Transfer เดียว 307KB > ESP32-S3 limit 32KB/transaction |
| 3 | สีบ้า | DMA อ่าน cached PSRAM sprite ได้ค่าผิด |
| 4 | แถบเรียง | คำสั่ง 0x2C ถูกกินรวมกับ data → GRAM pointer ไม่ reset |
| 5 | สีเพี้ยน (test) | Test code ส่ง LSB-first ขณะ renderer ส่ง MSB-first |
| 6 | ขาวตอน replug | Init อาศัย config ที่ firmware ก่อนหน้าทิ้งไว้ (parasitic) |
| 7 | โมเสกบาง replug | Init 40MHz ยิงเข้าจอที่ไฟยังไม่ stable → corrupt |
| 8 | ดำสนิท (pet ไม่โชว์) | จอต้องการ non-black frame ก่อนถึงจะ hold ภาพได้ |
| 9 | จอกระพริบ/กลับ | Phantom touch — parse ผิด ไม่มี range check |
| 10 | ภาพเล็ก | วาด 1:1 (96px) บนจอ 480px — ไม่ scale |
| 11 | Build flag ค้าง | Config จาก firmware เก่าหลุดเข้า build ใหม่ |
| 12 | "แก้แล้ว" ที่ไม่ได้แก้ | **เครื่องมือวัดมีจุดบอดเดียวกับบั๊ก** |

## ชั้นที่ 12 — บทเรียนที่แพงที่สุด

> "My measuring instrument shared the bug's blind spot."

Serial monitor ทุกตัวเปิด connection ด้วยการ toggle DTR = warm reset
จอ AXS15231 อยู่บน 3V3 rail ที่ไม่ดับตอน ESP32 reset
ดังนั้น serial log **ไม่มีวันเห็นบั๊ก cold-boot** — จอจำ config เก่าเสมอ

ผลคือ:
- Log บอก "frame pushed, 0 errors" → **ถูก** (data ถูกส่งจริง)
- จอดำ → **ถูกเหมือนกัน** (จอไม่ได้รับ init ที่ถูกต้องตอน cold boot)
- AI ประกาศ "fixed" **5 ครั้ง** โดยวัดจาก warm reset ทุกครั้ง
- มนุษย์ดึงปลั๊กแล้วจอดำทุกครั้ง — **มนุษย์ถูกทุกครั้ง**

เครื่องมือที่ทำให้เห็นความจริงคือ: ถอดปลั๊ก + ถ่ายรูปจอ + sample pixel buffer
ไม่ใช่ serial log

## Moral 3 ข้อ

### 1. อย่าเชื่อเครื่องมือที่ไม่ได้วัดบน failure path จริง

Serial log ที่ warm reset ไม่ได้พิสูจน์ว่า cold boot ทำงาน
ถ้ามนุษย์เห็นอย่างหนึ่ง แต่ log บอกอีกอย่าง — ถามว่า log **วัดตรงจุดที่พัง** หรือเปล่า
ถ้าไม่ → log กำลังโกหกคุณ

### 2. อาการเหมือนกัน ≠ สาเหตุเดียวกัน

"จอดำ" มีได้ 12 สาเหตุ แก้สาเหตุที่ 1 ไม่ได้แปลว่าจอจะติด —
แค่เปิดทางให้สาเหตุที่ 2 แสดงตัว หน้าตาเหมือนเดิมทุกประการ
อย่าท้อ — ทุกชั้นที่แก้ได้คือชั้นที่ไม่ต้องเจออีก

### 3. ตรวจสอบราคาถูกก่อนทำอะไรราคาแพง

Pattern ที่ซ้ำ 4 ใน 5 sessions ของมาราธอนนี้:
"ข้ามการตรวจสอบง่ายๆ ก่อนลงมือทำสิ่งที่กินเวลา"
— ไม่ดู silkscreen ก่อน flash
— ไม่ run `esphome config` ก่อนประกาศ verdict
— เปลี่ยน 2 ตัวแปรพร้อมกันแทนที่จะ isolate
— ไม่ search ว่าครอบครัวเคยทำเรื่องนี้แล้วหรือยัง

คำสั่งเดียวที่ถูกต้อง ใช้เวลา 5 วินาที ป้องกันการเสียเวลาหลายชั่วโมง

## ทำไมคอร์สเรียงแบบนี้

| ระดับ | สิ่งที่ได้ | ทำไมต้องมาก่อน |
|-------|-----------|---------------|
| **Basic** | จอติด + pixel แรก | พิสูจน์ว่าฮาร์ดแวร์ทำงาน — ก่อนจะ debug อะไรทั้งนั้น ต้องรู้ว่าจอไม่ได้พัง |
| **Standard** | จอขยับ + ตัวเลขเปลี่ยน | เข้าใจว่า display lambda = ฟังก์ชันของ state — ไม่ใช่รูปวาดครั้งเดียว |
| **Intermediate** | LVGL + touch + ปุ่มกด | ก้าวข้าม lambda ไป widget system — จอ "โต้ตอบ" ได้จริง |
| **Advanced** | (ยังไม่ถึง) | ทุกอย่างข้างบนรวมกัน + ความรู้จากหอคอย 12 ชั้น |

ลำดับนี้ไม่ได้เรียงตามความยาก — เรียงตาม **ขอบเขตที่คุณต้อง debug**:
- Basic พัง = ฮาร์ดแวร์หรือ pin ผิด (ตรวจง่าย)
- Standard พัง = logic หรือ timing (ดู serial ได้)
- Intermediate พัง = LVGL state หรือ event routing (ซับซ้อนขึ้น)
- Advanced พัง = ทุกชั้นข้างบน + bus/DMA/power/init (**ต้องรู้ว่าปัญหาอยู่ชั้นไหน**)

ถ้าคุณกระโดดไป Advanced วันแรก คุณจะเจอ "จอดำ" แล้วไม่รู้ว่ามันดำเพราะชั้นไหน
ถ้าคุณเรียง Basic → Standard → Intermediate ก่อน
คุณจะรู้ว่า "จอเคยติดที่ระดับ X" = ปัญหาอยู่เหนือ X ขึ้นไป

นั่นคือพลังของการเรียงลำดับ: **ทุกระดับที่ผ่านแล้วคือ checkpoint ที่ตัดสาเหตุออกไปทั้งชั้น**
