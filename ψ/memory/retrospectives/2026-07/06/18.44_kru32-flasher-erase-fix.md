## Honest Feedback

1. **GitHub Pages deploy flakiness ate a disproportionate number of turns.** The first Pages setup (23:29–00:10) needed a `.nojekyll` fix after an hour-long stuck build; the Van Gogh deploy (15:28–15:44) needed multiple retries the AI first suspected was rate-limiting before confirming it was just backend provisioning lag; the self-host deploy (16:33–16:34) failed twice more. None of it was our config, but "is my fix even live yet" was a recurring distraction layered on top of an already-hard debugging session.

2. **Improv Serial was added to all 16 lessons before the crash cause was understood**, in the middle of open, unresolved crash reports. It made the crash *more* reproducible (post-flash serial reconnect racing board reboot), forced a full revert to a byte-identical firmware baseline, then had to be re-implemented a second time with an added mitigation flag once WiFi config was needed again — two full recompiles of 16 lessons for one feature.

3. **Self-hosting esp-web-tools to drop the CDN (16:25–16:34) shipped an incomplete vendored copy** (missing dynamic-import chunks), producing a brand-new failure mode — a silent, do-nothing flash — stacked on top of the still-unresolved crash. The user's angriest message in the session landed right after this, while two different bug classes (crash vs. silence) were being debugged at once.

## Lessons Learned

1. **When a working reference implementation exists (same org, same hardware, same problem class), diff against it FIRST** — before generating independent theories. `workshop-04-esp32-wasm` was one repo away the entire time; a manifest diff against it would have surfaced `erase:false` in under a minute instead of ~40 minutes of independent diagnosis.

2. **A deep-research/subagent finding you commissioned is only useful if you act on its top recommendation.** Research that names the trigger explicitly ("native USB_SERIAL_JTAG + erase = crash 100%") should change the very next line of config touched — not get filed as supporting evidence for a different fix already decided on.

3. **For hardware/native crashes (WebSerial, WebUSB, USB re-enumeration, driver-level bugs), isolate ONE variable at a time in a minimal repro before shipping a fix.** Bundling a version bump + a self-host/vendoring change + a config flag into one deploy makes it impossible to know which change fixed — or broke — anything. Two unrelated bugs (a crash and a silent failure) got debugged simultaneously here because of exactly this.

4. **Read the crash report's `Process:` / `Identifier:` line before attributing a native crash to a specific app.** It's the first thing in every macOS crash log (`Comet [85052]` / `ai.perplexity.comet` vs `Google Chrome [33118]` / `com.google.Chrome`) — skipping straight to the stack trace led directly to a confidently wrong diagnosis here.
