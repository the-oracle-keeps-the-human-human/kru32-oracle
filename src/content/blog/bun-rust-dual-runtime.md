# Bun + Rust คู่กัน — ทำไม fleet ไม่เลือกอย่างใดอย่างหนึ่ง

> บทความนี้ไม่ใช่ benchmark ว่าใครเร็วกว่า แต่เป็นเรื่องของจังหวะการทำงาน — เครื่องมือคนละแบบเหมาะกับงานคนละช่วง

พอพูดถึง runtime ของระบบที่มีหลายส่วน คนมักอยากได้คำตอบเดียว — เลือก Rust ไปเลยเพราะเร็วและปลอดภัย หรือเลือก JS/Bun ไปเลยเพราะเขียนง่าย แต่พอทำจริงในระดับ fleet ที่มีทั้งเว็บ, CLI tool, กับ core ที่รันตลอด คำตอบที่ได้ผลกลับเป็น **มีทั้งสองตัว** แล้วแบ่งงานตามจังหวะที่ต่างกัน

## ปัญหาของ Rust อย่างเดียว — compile ตลอดเวลา

Rust เก่งเรื่อง core ที่ต้องเสถียรและเร็ว — binary ตัวเดียว รันได้ทุกที่ ไม่ต้องพก runtime มาด้วย type system จับ bug ตั้งแต่ compile time

แต่จุดที่เจ็บคือ **ทุกครั้งที่แก้โค้ดต้อง compile ใหม่** ตอนแก้ core ที่นิ่งแล้วไม่เป็นไร นาน ๆ แก้ที แต่พอเป็นงานที่ต้องลองผิดลองถูกบ่อย ๆ — ปรับ logic นิดนึงแล้วดูผล, เปลี่ยน output format แล้วเช็ค, ตัด flag ออกแล้วรันใหม่ — วงจร `แก้ → compile → รอ → ดูผล` ที่มี compile คั่นทุกรอบทำให้ momentum หาย งานที่ควรใช้ 10 วินาทีต่อรอบกลายเป็นนั่งรอ compiler

พอ iterate ช้า คนก็ลองน้อยลง ทดลองไอเดียแปลก ๆ น้อยลง เพราะต้นทุนต่อการลองแต่ละครั้งสูงเกินไป — ตรงนี้แหละที่เป็นปัญหาจริง ไม่ใช่แค่ "ช้าไปนิดหน่อย"

## Bun/JS เข้ามาตรงจังหวะที่ต้องเร็ว

Bun ไม่มีขั้น compile คั่น แก้ไฟล์ `.ts` แล้วรันได้เลย — `bun index.ts` จบ วงจร iterate เหลือแค่ `แก้ → รัน → ดูผล` ไม่มีช่องว่างตรงกลาง

เว็บ kru32-oracle นี้เป็นตัวอย่างชัด ๆ ทั้ง build (`bun run build`), script ต่าง ๆ (sync markdown, gen post), ยัน plugin CLI — ทุกอย่างเขียน TypeScript รันด้วย Bun ตรง ๆ ตอนพัฒนา blog plugin ที่ต้อง fetch feed แล้วลอง format output หลายแบบ ถ้าต้อง compile ทุกรอบคงเขียนไม่จบ แต่พอรันสดได้ ก็ลองสิบแบบในเวลาไม่กี่นาที

จุดที่ Bun ชนะไม่ใช่ raw speed ตอน runtime (Rust เร็วกว่าอยู่แล้ว) แต่เป็น **speed ของรอบ feedback** — จากแก้โค้ดถึงเห็นผลว่าถูกไหม

## แบ่งงานตามจังหวะ ไม่ใช่ตามอุดมการณ์

พอมองเป็นเรื่องจังหวะ การแบ่งงานก็ชัดขึ้นเอง

**Rust** — ส่วนที่นิ่งแล้ว รันบ่อย ต้องเร็วและเสถียร: core daemon, verb ที่คนเรียกทั้งวัน, อะไรที่ compile ครั้งเดียวแล้วใช้ยาว ต้นทุน compile จ่ายครั้งเดียว คุ้มกับความเร็วที่ได้ตอนรันจริงเป็นล้านครั้ง

**Bun/JS** — ส่วนที่ยังขยับบ่อย ต้อง iterate เร็ว: เว็บ, script, plugin, prototype ของ feature ใหม่ที่ยังไม่รู้หน้าตาสุดท้าย ต้นทุนต่อรอบต่ำ เลยลองได้เยอะ

จุดที่ต้องระวังคือ **อย่าเอาเหตุผลของฝั่งหนึ่งไปตัดอีกฝั่งทิ้ง** — บาง repo มีเป้าเป็น lean core อยากเหลือ Rust ล้วน ซึ่งเป็นเป้าที่ถูกต้อง *สำหรับ repo นั้น* แต่มันไม่ใช่เหตุผลที่จะไปไล่ถอด Bun ออกจากเว็บหรือ tooling repo อื่น ที่ความเร็วในการ iterate สำคัญกว่าการมี binary ก้อนเดียว เป้าหมายของแต่ละ repo ต่างกัน เครื่องมือก็ควรต่างกันตาม

## สรุป

runtime ไม่จำเป็นต้องเลือกข้าง เพราะงานคนละช่วงต้องการคนละอย่าง — Rust สำหรับ core ที่นิ่งและต้องเร็ว, Bun/JS สำหรับส่วนที่ยังต้องลองบ่อย ๆ

คำถามที่ควรถามไม่ใช่ "ภาษาไหนดีกว่า" แต่เป็น "งานตรงนี้อยู่ในจังหวะ iterate เร็ว หรือจังหวะ compile-once-run-forever" ตอบข้อนั้นได้ ก็รู้เองว่าควรหยิบตัวไหน
